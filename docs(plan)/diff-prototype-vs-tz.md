# Diff: прототип `queue_prot_blance` vs ТЗ «Load Balancer for Telegram Channels»

Документ фиксирует различия между текущей реализацией в репозитории и техническим заданием из файла `Load Balancer for Telegram Channels (1).docx`.

**Дата:** 2026-06-06  
**Прототип:** `queue_prot_blance` (Huey + Redis + MockTelegramClump)  
**ТЗ:** балансировщик нагрузки для работы с Telegram-каналами через аккаунты (PostgreSQL)

---

## Сводка в одном абзаце

Прототип решает задачу **балансировки HTTP-запросов discovery API** и **выбора Telethon-сессии на каждую операцию** внутри пайплайна. ТЗ описывает **доменную очередь задач по каналам** с персистентным хранением в PostgreSQL, полным жизненным циклом задач, учётом ресурса аккаунтов и фоновыми продюсерами. Архитектуры пересекаются по идеям (приоритеты, least-loaded, лимиты), но работают на **разных уровнях** и с **разными единицами работы**.

---

## 1. Назначение и границы системы

| Аспект | Прототип | ТЗ |
|--------|----------|-----|
| Основная задача | Распределить нагрузку HTTP-эндпойнтов discovery API между Telethon-сессиями | Распределить доменные задачи по каналам между Telegram-аккаунтами |
| Контур поиска | Внутри очереди: эндпойнты `discover`, `discover-groups` с поисковыми ops | Явно **вне** балансировщика; балансировщик работает с каналами уже в БД |
| Связь с каналами | Нет модели владения каналом аккаунтом | Центральная: один канал → один аккаунт, перенос, выравнивание ±5% |
| Кто создаёт задачи | Внешний HTTP-клиент через mock API (`enqueue_endpoint`) | Фоновые продюсеры + другие инструменты в общую `task_queue` |

### Вывод

Прототип — **слой исполнения API-запросов**. ТЗ — **слой управления жизненным циклом каналов** (обновление, сбор данных, перенос между аккаунтами).

---

## 2. Архитектура

### Прототип: двухуровневая

```
HTTP-запрос
    → Уровень 1: Huey / Redis (приоритет эндпойнта)
        → Worker забирает задачу
            → Уровень 2: Clump / сессии (выбор аккаунта на каждую op)
                → Telethon-операция (mock)
```

Ключевые файлы: `queue_reg.py`, `tasks.py`, `priorities.py`, `classes.py`, `ops_catalog.py`.

### ТЗ: одноуровневая доменная очередь

```
Продюсер (channel_balancer / collect_extra_data / update_channel)
    → PostgreSQL task_queue
        → Балансировщик-воркер (выбор задачи + выбор аккаунта + жизненный цикл)
            → Исполнитель задачи (Telethon)
```

### Diff

| | Прототип | ТЗ |
|---|----------|-----|
| Уровней балансировки | 2 (очередь Redis + выбор сессии) | 1 (очередь + балансировщик в одном контуре) |
| Разделение «очередь / аккаунт» | Разные шкалы приоритетов (Huey 10/5/1 и Balance 8/5/3) | Один приоритет задачи + настройки в `task_types` |
| Балансировка между воркерами Huey | Да (порядок dequeue) | Нет (балансировка только между аккаунтами) |

## 3. Единица работы (критическое расхождение)

### ТЗ

> Одна задача в очереди = одна единица работы с аккаунтом.  
> 500 сообщений → 500 отдельных задач, не одна пакетная.

Примеры задач: обновить один канал, собрать одну единицу данных, один перенос.

### Прототип

Один HTTP-запрос = **одна задача Huey** на **весь пайплайн** эндпойнта:

```python
# tasks.py
run_endpoint_pipeline(endpoint_key, payload, huey_priority)
```

Внутри — последовательный цикл по всем ops из `ops_catalog.py`. Например, `POST /discovery-api/discover` включает 10+ операций (`connect`, `contacts.Search`, `messages.SearchGlobal`, …) в **одной** задаче Redis.

### Diff

| | Прототип | ТЗ |
|---|----------|-----|
| Гранулярность | 1 HTTP-запрос = 1 задача = N ops | 1 действие = 1 задача = 1 аккаунт |
| Пакетные действия | Допускаются (пайплайн эндпойнта) | Запрещены |
| Параллельность ops внутри запроса | Нет (строго последовательно) | Не применимо (ops не объединяются) |

## 4. Хранилище и персистентность

| Компонент | Прототип | ТЗ |
|-----------|----------|-----|
| Очередь | Redis, `PriorityRedisHuey` (`queue_reg.py`) | PostgreSQL `task_queue` |
| Аккаунты | Mock в памяти (`clump_registry.py`, `MockTelegramClump`) | Таблица `accounts` |
| Типы задач / настройки | Хардкод в `priorities.py`, `classes.py` | Таблица `task_types` |
| Учёт ресурса | `MockTelegramparser.current_request_count` (in-memory) | `account_resource_usage` (за последний час) |
| История попыток | Только логи | Таблица `task_attempts` |
| Каналы | Не используются | Существующая таблица каналов + связи |

### Последствия

- После рестарта воркера счётчики RPH в прототипе **обнуляются**.
- В ТЗ учёт попыток и ресурса **переживает перезапуски**.
- В ТЗ все лимиты и размеры очереди — **настройки в БД**, не в коде.

## 5. Модель аккаунта

### ТЗ (`accounts`)

- Поля: `status`, `hourly_limit`, `is_enabled`, `cooldown_until`, `current_task_id`, `last_used_at`, `last_error`, …
- Статусы: `active`, `cooldown`, `disabled`, `banned`, `error`
- **Один аккаунт — не более одной задачи одновременно** (`current_task_id`)
- Cooldown после flood_wait и подобных ошибок

### Прототип (`MockTelegramparser` / `MockTelegramClump`)

- Три фиксированные сессии: `session_alpha`, `session_beta`, `session_gamma`
- Нет статусов, cooldown, `current_task_id`
- Нет привязки каналов к аккаунту (`channels_list` — mock-поле)
- Несколько воркеров Huey могут **одновременно** использовать одну сессию в разных задачах

### Diff

| | Прототип | ТЗ |
|---|----------|-----|
| Статусы аккаунта | Нет | Обязательны |
| Блокировка «аккаунт занят» | Нет | Да (`current_task_id`) |
| Параллельность на аккаунт | Неограничена (N воркеров × M задач) | Макс. 1 задача |
| Cooldown / ban | Нет | Да |

## 6. Модель ресурса и лимитов

### ТЗ

- **Единый** почасовой лимит на аккаунт: `hourly_limit` (например, 100/час)
- Формула: `available_resource = hourly_limit - used_attempts_last_hour`
- Считаются **все попытки** за последний час, включая неуспешные
- Минимальный свободный ресурс для запуска: `min_available_resource_percent` в `task_types`
- Расход фиксируется в момент передачи задачи аккаунту в работу
- Проверка балансировщиком без запуска — **не** расход

### Прототип

- Лимиты **по типу Telethon-операции** (`rph_limits` в `classes.py`):

  | task_type | Лимит RPH |
  |-----------|-----------|
  | `contacts.Search` | 2 |
  | `messages.SearchGlobal` | 120 |
  | `channels.JoinChannel` | 30 |
  | `get_entity` | 7 |
  | … | … |

- Эффективный лимит: `rph_limits[task_type] × (1 - reserve)`, `reserve = 0.1`
- Счётчик увеличивается при каждом `mock_execute` (успешном)
- При перегрузе всех сессий ops с `balance_priority >= threshold_priority` (8) могут **пробить** лимит
- Нет `hourly_limit` на аккаунт, нет процента свободного ресурса

### Diff

| | Прототип | ТЗ |
|---|----------|-----|
| Единица лимита | По типу Telethon-op | По аккаунту (час) |
| Учёт неуспешных попыток | Да (счётчик при execute) | Да (в `account_resource_usage`) |
| «Пробитие» лимита высоким приоритетом | Да (`threshold_priority = 8`) | Нет (жёсткий `min_available_resource_percent`) |
| Персистентность учёта | In-memory | PostgreSQL |
| Задачи с двумя аккаунтами | Не поддерживаются | `source_resource_cost` + `target_resource_cost` |

## 7. Приоритеты

### ТЗ

- Числовая шкала в БД (пример: 1000 срочные, 500 важные, 200 сбор данных, 100 балансировка, 50 обновление)
- Настраивается в `task_types.default_priority` и `task_queue.priority`
- Сортировка: приоритет ↓, затем `created_at` ↑
- **Не хардкодить** лимиты и размеры очереди в коде

### Прототип

Две **захардкоженные** шкалы в `priorities.py`:

**Huey (очередь Redis):**

| Уровень | Значение | Эндпойнты |
|---------|----------|-----------|
| HIGH | 10 | add-channel, parser/add-channels |
| NORMAL | 5 | auth, parser/start, bot/send-message, … |
| LOW | 1 | discover, discover-groups |

**Balance (выбор сессии):**

| Уровень | Значение | Условие |
|---------|----------|---------|
| HIGH | 8 | `ADD_CHANNEL_OPS` или HIGH-эндпойнт |
| LOW | 3 | `SEARCH_OPS` или LOW-эндпойнт |
| NORMAL | 5 | остальное |

### Diff

| | Прототип | ТЗ |
|---|----------|-----|
| Где задаётся | Код (`priorities.py`) | БД (`task_types`) |
| Шкала | 1 / 5 / 10 и 3 / 5 / 8 | 50–1000+ |
| Привязка | К HTTP-эндпойнту и Telethon-op | К доменному типу задачи |
| Две шкалы | Да (Huey + Balance) | Одна (priority задачи) |

## 8. Жизненный цикл задачи

### ТЗ

Статусы: `queued`, `scheduled`, `in_progress`, `retry`, `done`, `failed`, `cancelled`, `stuck`.

Механизмы:

- `run_after` — отложенный запуск
- `attempt_count` — реальные попытки через аккаунт (расход ресурса)
- `postpone_count` — откладывания без расхода ресурса
- `dedup_key` — защита от дублей активных задач
- `locked_by`, `locked_at`, `locked_until` — атомарное взятие в работу
- Retry: `max_attempts`, `retry_delay_seconds`, `retry_backoff_multiplier`, `max_retry_delay_seconds`
- Watchdog: `task_timeout_seconds` → `stuck` / `retry`

### Прототип

```
ENQUEUE → Redis → DEQUEUE → PIPELINE (ops по порядку) → DONE
```

- Статусов задачи нет (только `task_id` Huey)
- При отказе балансировки: op → `ok=False`, **пайплайн продолжается**
- Retry, postpone, dedup, stuck — **отсутствуют**
- Атомарность: Huey гарантирует один dequeue на задачу, но нет row-level lock

### Diff

| Механизм | Прототип | ТЗ |
|----------|----------|-----|
| Статусы задачи | Нет | 8 статусов |
| `run_after` | Нет | Да |
| `attempt_count` / `postpone_count` | Нет | Да, раздельно |
| `dedup_key` | Нет | Да |
| Retry с backoff | Нет | Да |
| Обработка зависших | Нет | Да (`stuck`, timeout) |
| Отложить и взять следующую задачу | Нет (воркер занят пайплайном) | Да |

---

## 9. Алгоритм выбора задачи

### ТЗ

1. Фильтр: `queued` / `retry`, `run_after <= now`, тип включён, не заблокирована, `attempt_count < max_attempts`
2. Сортировка: priority DESC, created_at ASC
3. Если задача не может выполниться сейчас → postpone, `postpone_count++`, перейти к **следующей** задаче

### Прототип

1. Huey dequeue по priority (10 > 5 > 1)
2. Воркер выполняет **весь** пайплайн эндпойнта до конца
3. Проблемная op не блокирует очередь Redis, но **блокирует воркер** на время пайплайна

### Diff

Прототип не реализует сценарий «одна проблемная задача не должна стопорить всю очередь» на уровне доменных задач — воркер занят текущим пайплайном, следующая задача из Redis не обрабатывается параллельно внутри того же воркера.

---

## 10. Алгоритм выбора аккаунта

### Общее (пересечение)

Оба варианта используют **least-loaded**: выбирается наименее нагруженный подходящий аккаунт.

**Прототип** (`balance_load` в `classes.py`):

1. Меньше счётчик по данному `task_type`
2. При равенстве — меньше суммарная нагрузка
3. При равенстве — лексикографически `session_name`

**ТЗ:**

1. Аккаунт не выполняет другую задачу
2. Достаточно ресурса и `min_available_resource_percent`
3. Меньше задач за последний час
4. Выше процент свободного ресурса

### Diff

| | Прототип | ТЗ |
|---|----------|-----|
| Задача с фиксированным аккаунтом | Нет (`account_id` в задаче) | Да (`account_id` в `task_queue`) |
| Проверка cooldown / status | Нет | Да |
| Проверка «аккаунт свободен» | Нет | Да (`current_task_id IS NULL`) |
| Задачи с двумя аккаунтами | Нет | Да (`source_account_id`, `target_account_id`) |
| Откладывание при недоступности | Отказ op (`ok=False`) | Postpone задачи с `run_after` |

## 12. Инструменты-создатели задач

### ТЗ

Три встроенных низкоприоритетных продюсера + расширяемость:

1. **Балансировщик каналов** — перенос для выравнивания ±5%
2. **Сбор доп. данных** — каналы без собранных данных
3. **Обновление старых каналов** — фоновое обновление

Общие правила: `task_queue`, `task_types`, `dedup_key`, `target_queue_size`, без хардкода лимитов.

### Прототип

Задачи создаёт только **внешний HTTP-клиент** через mock API (`mock_api.py` → `enqueue_endpoint`). Фоновых продюсеров нет. Нагрузочная симуляция — `demo_load.py`.

- Метрик, алертов, watchdog зависших задач — **нет**

## 14. Обработка ошибок

### ТЗ

Классификация: временная → retry; ресурс → postpone; аккаунт → cooldown; постоянная → failed; данные → failed / ручная проверка.

Обязательное логирование: task_id, тип, аккаунт, канал, код ошибки, номер попытки, будет ли повтор.

### Прототип

- Исключение в Huey-задаче → лог `DONE` с `error`
- Отказ балансировки на op → `ok=False`, `reason`, пайплайн продолжается
- Классификации ошибок, cooldown аккаунта, retry — **нет**

## 18. Возможная стратегия сближения (справочно)

Прототип можно не выбрасывать, а разделить слои:

```
[ТЗ] Продюсеры → task_queue (PostgreSQL) → балансировщик-воркер
                                              ↓
[Прототип] При необходимости: enqueue в Huey или прямой вызов clump/Telethon
```

- **Слой ТЗ:** PostgreSQL, жизненный цикл, продюсеры, мониторинг, домен каналов.
- **Слой прототипа:** исполнение Telethon-ops с `balance_load` и RPH-лимитами (доработать под `accounts` из БД).

Критические доработки прототипа для соответствия ТЗ:

1. Заменить in-memory счётчики на `account_resource_usage`
2. Добавить статусы аккаунтов и `current_task_id` (1 задача на аккаунт)
3. Разбить пайплайны на атомарные задачи в `task_queue`
4. Реализовать retry, postpone, dedup, мониторинг
