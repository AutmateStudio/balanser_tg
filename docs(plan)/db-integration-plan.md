# Объединение БД балансировщика и БД лидогенерации (текущее исполнение)

**Дата:** 2026-06-13
**Контекст:** есть две схемы —
- `DB/Lidogen_main_DB.sql` — продуктовая БД платформы лидогенерации (владелец `lead_monitor_owner`): `platforms`, `source_channels`, `source_messages`, `monitoring_projects`, `project_source_channels`, биллинг, пользователи и т.д.
- `DB/BD_schema.sql` — greenfield-схема PG Queue Balancer: `accounts`, `task_types`, `resource_op_types`, `task_type_ops`, `task_queue`, `task_attempts`, `account_resource_usage`, упрощённая `source_channels`, мониторинговые view (§26).

Ниже — как их свести в **одной** PostgreSQL-БД без дублирования и потери данных.

---

## 1. Узел конфликта — `source_channels`

| Аспект | main DB | greenfield `BD_schema.sql` |
|--------|---------|----------------------------|
| Идентичность канала | `(platform_id, external_channel_id)` | `channel_ref` UNIQUE |
| `id` | `GENERATED ALWAYS AS IDENTITY` | `GENERATED BY DEFAULT AS IDENTITY` |
| Балансировочные поля | нет | `assigned_account_id`, `extra_data_collected`, `last_updated_at` |
| Платформенные поля | `platform_id`, `metadata`, `linked_projects_count`, триггеры | нет |

`source_channels` в main DB — **источник истины по каналам**. Дублировать её в очереди нельзя: разойдутся `source_messages`, `project_source_channels` и триггеры пересчёта.

**Решение:** очередь использует существующую `source_channels`, лишь **расширяя** её балансировочными колонками (ТЗ §5.1, задача A8). `task_queue.channel_id` ссылается на `source_channels.id` (тот же `int8`).

---

## 2. Рекомендуемый режим — единая БД (Shared DB)

Очередь и платформа живут в одной БД (`QUEUE_DATABASE_URL == DATABASE_URL`).

**Плюсы:**
- `task_queue.channel_id → source_channels.id` — честный FK, без cross-DB sync.
- Продюсеры (F2/F4/F5) читают каналы напрямую (балансировка ±5%, `last_updated_at`, `extra_data_collected`).
- Discovery (`channel_store.py`) уже пишет в эту `source_channels` — данные сразу доступны воркеру.
- Одна транзакционная граница, одни бэкапы.

**Минусы / риски:**
- Нагрузка воркера (claim в цикле, `SKIP LOCKED`) делит ресурсы с продуктовой БД → отдельный пул соединений, ограничить `max_connections`.
- Имена объектов не должны конфликтовать (проверено: таблицы очереди уникальны).
- Права: воркеру выдать доступ только к таблицам очереди + `SELECT/UPDATE` нужных колонок `source_channels`.

### Как накатывать

Использовать **`DB/A8_integrate_main_db.sql`** (создан рядом), а НЕ `BD_schema.sql`:

```bash
# поверх существующей Lidogen_main_DB
psql "$QUEUE_DATABASE_URL" -f DB/A8_integrate_main_db.sql
psql "$QUEUE_DATABASE_URL" -f DB/A9_seed.sql
```

Файл идемпотентен (`IF NOT EXISTS` / `DO`-guard) и:
1. создаёт ENUM-типы и таблицы очереди (без `platforms` и `source_channels`);
2. делает `ALTER TABLE source_channels ADD COLUMN ... assigned_account_id / extra_data_collected / last_updated_at`;
3. вешает FK `source_channels.assigned_account_id → accounts.id` и `task_queue.channel_id → source_channels.id`;
4. переиспользует существующую функцию `public.update_updated_at()` для триггеров новых таблиц (не дублирует триггер на `source_channels`);
5. создаёт мониторинговые view (§26).

> `BD_schema.sql` остаётся для **изолированного dev/CI** (greenfield), где main DB нет. Там он сам создаёт `platforms` (`IF NOT EXISTS`) и полноценную `source_channels`.

## 3. Альтернатива — раздельные БД (Separate DB)

Очередь в своей БД (`QUEUE_DATABASE_URL ≠ DATABASE_URL`), main DB отдельно.

**Когда оправдано:** разные SLA/нагрузка, отдельный сервер под воркер, нежелание давать воркеру доступ к продуктовым данным.

**Цена:**
- FK `task_queue.channel_id → source_channels.id` **невозможен** между БД → `channel_id` хранит ID из main DB как «слабую» ссылку (без FK), целостность — на стороне приложения.
- Нужен механизм синхронизации каналов: либо периодический ETL `source_channels → queue.source_channels`, либо postgres_fdw / логическая репликация только таблицы каналов.
- Dual-write (D7) и продюсеры усложняются, появляется лаг.

Для текущего этапа (MVP, discovery уже пишет в main DB) — **избыточно**. Рекомендация: режим 2 (Shared DB).

## 4. Сопоставление понятий при интеграции

| Балансировщик / discovery | main DB |
|---------------------------|---------|
| `channel_ref` (`@channel`, t.me/…) | `external_url`, либо username из `metadata` |
| Telegram peer id | `external_channel_id` |
| «канал на parser-сессии» | `source_channels.assigned_account_id` (новая колонка) |
| «собраны extra data» | `source_channels.extra_data_collected` (новая колонка) |
| «пора update_channel» | `source_channels.last_updated_at` (новая колонка; ≠ `updated_at`) |
| «активная задача по каналу» | `task_queue WHERE channel_id = … AND status IN (...)` |
| «канал в проекте» | `project_source_channels` (не трогаем) |

> `last_updated_at` — отдельная колонка для продюсера F5; общий аудитный `updated_at` ведёт триггер `trg_source_channels_update_updated_at` и его перезаписывать нельзя.

## 5. Порядок внедрения (чек-лист)

1. Решение зафиксировано: **Shared DB** (один PostgreSQL).
2. `QUEUE_DATABASE_URL` указывает на ту же БД, что и продуктовый `DATABASE_URL` (A1/Z3 — `.env.example`).
3. Накат: `A8_integrate_main_db.sql` → `A9_seed.sql` (через runner A11 / `make migrate-queue`).
4. Seed `platforms` не нужен — берётся существующий (`code='tg'`), discovery его уже использует.
5. A10: sync `accounts` из SQLite `account_store` / clump → таблица `accounts` очереди.
6. Проверка: `task_queue.channel_id` резолвится в `source_channels`; воркер видит `assigned_account_id`.
7. Отдельный пул соединений и ограничение конкурентности воркера, чтобы не мешать продуктовой нагрузке.
