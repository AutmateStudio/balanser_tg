# Очередь в PostgreSQL: что сделано и куда идём

**Формат:** связный рассказ словами (без таблиц и карточек)  
**Дата:** 2026-06-18  
**Для кого:** команда, заказчик, любой, кто хочет понять проект «с высоты птичьего полёта»  
**Техническая версия:** `docs(plan)/рассказ-спринты-s1-s2-s3.md`

---

## Зачем вообще всё это

Сегодня балансировщик Telegram-аккаунтов в discovery работает по-старому: пришёл запрос «добавить каналы» — и вся пачка уходит в SQLite-очередь `action_queue`, откуда SessionClump забирает её одним bulk-потоком. Нет атомарности по каналам, нет нормальных приоритетов, нет защиты от дублей, нет осмысленного retry и postpone, нельзя безопасно запустить несколько воркеров параллельно.

Техническое задание описывает другую модель: **доменная очередь в PostgreSQL**. Каждый канал — отдельная задача. У задачи есть тип, приоритет, dedup-ключ, время отложенного запуска. Аккаунты резервируются на время исполнения. Лимиты Telegram считаются не «одним числом на аккаунт», а **по каждой Telethon-операции отдельно** — join, resolve, чтение сообщений и так далай. Несколько воркеров могут работать одновременно и не схватят одну и ту же задачу.

Именно этот переход мы сейчас строим. Дорожная карта разбита на пять спринтов. Первый уже позади. Второй начат и перетекает в третий. Остальное — retry, продюсеры, мониторинг — впереди.

## Что мы сделали за прошлый спринт

Между девятым и семнадцатым июня команда закрыла первый спринт целиком и успела заложить начало второго. По карточкам ТЗ это двадцать четыре задачи — весь блок схемы PostgreSQL, почти все репозитории, конфигурация, sync аккаунтов, процесс воркера в Docker и первые методы SessionClump, которые понадобятся воркеру завтра.

Если говорить образно: **мы построили фундамент, стены и крышу**. Внутри уже есть лифтовая шахта, электрический щит и пожарная сигнализация — но лампочки в квартирах ещё не зажгли. Задачи в очередь попадают, воркер их забирает, но пока «исполняет» заглушкой: сразу помечает выполненными, не трогая Telethon. Это осознанный scope первого этапа.

### От bulk к платформе

Раньше цепочка была простой и хрупкой: API кладёт bulk в SQLite, clump тащит всё подряд. Теперь в центре — PostgreSQL. Туда API и будущие продюсеры будут класть атомарные задачи. Оттуда отдельный процесс-воркер их забирает по правилам приоритета и блокировок. А SessionClump остаётся исполнителем Telethon — но уже не сам решает, *какой* аккаунт взять: это решает очередь.

Одно из важнейших решений спринта — **per-op учёт лимитов**. В исходном ТЗ лимит был одним числом на аккаунт. Мы пошли глубже: завели каталог Telethon-операций — resolve, join, leave, чтение сообщений, поиск — у каждой свой лимит запросов в час и свой резерв. Задача в очереди описывает, какие операции она потратит и в каком количестве. Для переноса канала между аккаунтами есть роли «исходный» и «целевой» — лимиты проверяются на обоих.

Это не академическая деталь. Telegram бьёт по flood именно на конкретных методах. Когда воркер дойдёт до проверки ресурса, он сможет сказать: «у этого аккаунта join ещё можно, но resolve уже на исходе — задачу лучше отложить», а не гадать по одному общему счётчику.

### База данных: не миграция «на будущее», а рабочая модель

Мы написали полную DDL-схему — порядка пятисот сорока строк SQL. В ней восемь таблиц, четыре enum-типа для статусов аккаунтов, задач и попыток, больше двадцати индексов и восемь представлений для мониторинга.

Статусы задачи проходят весь жизненный цикл: поставлена, отложена, в работе, retry, выполнена, провалена, застряла. Статусы аккаунта — активен, на cooldown после flood, отключён, забанен, ошибка. Вся политика retry, postpone, таймаутов и порогов ресурса лежит в таблице типов задач — не в переменных окружения и не в коде clump. Хотим поменять приоритет add-channel с пятисот на триста — правим seed, а не деплоим код.

Очередь задач защищена от дублей: partial unique index не даёт создать вторую *активную* задачу с тем же dedup-ключом. Диспетчерский индекс ускоряет выборку готовых задач. Отдельные индексы заложены под watchdog для зависших задач и под алерты по частым postpone — мы их ещё не включаем, но схема уже готова.

Параллельно с DDL мы заполнили справочники. В seed попали девятнадцать Telethon-операций с лимитами, снятыми из реального кода discovery — от редких auth-операций до массового GetParticipant. Пять типов задач: add-channel и move-channel включены для MVP, collect и update выключены до продюсеров, remove-channel лежит в seed на будущее. Для add-channel и move-channel прописаны связки «тип задачи → какие op расходует» — без этого воркер не сможет ни проверить ресурс, ни списать usage.

Развёртывание схемы — одна команда через Makefile и shell-скрипт. Локально поднимается Docker с PostgreSQL 16, миграции накатываются автоматически, можно сразу запустить воркер против локальной базы. Для staging и prod — тот же скрипт, тот же DSN в `.env`. Режим Shared DB: очередь живёт в той же PostgreSQL, что и основная платформа Lidogen — отдельный сервер не нужен.

### Python-слой: репозитории, которые уже можно доверять production

Поверх схемы мы построили typed async-слой на asyncpg — семь модулей плюс entrypoint воркера.

Слой подключения даёт пул соединений, транзакции с автоматическим commit и rollback, healthcheck для probes. Есть smoke-тест, который доказывает: если внутри транзакции случилась ошибка, INSERT откатывается.

Репозиторий типов задач читает не только код и приоритет, но и полный состав операций — с ролями для dual-account, с лимитами RPH и резервом. Enqueue создаёт задачу, подставляет priority и max_attempts из справочника, и при повторе с тем же dedup возвращает «уже есть», а не плодит дубликаты.

Claim — сердце диспетчера. SQL находит среди готовых задач максимальный приоритет, внутри уровня выбирает случайную — так мы разгрузили спорный пункт ТЗ про «строгую сортировку vs random». Блокировка через `FOR UPDATE SKIP LOCKED`: два воркера никогда не возьмут одну строку. Задача с будущим `run_after` не трогается. Истёкший lock — задача снова доступна. Успех переводит в done, ошибка — в retry или failed по числу попыток.

Репозиторий аккаунтов умеет выбрать свободный, зарезервировать под задачу, освободить после execute. Самое важное — атомарный pick-and-reserve в одной транзакции с той же SKIP LOCKED семантикой: два воркера не получат один Telegram-аккаунт. Выбор идёт с учётом cooldown — задел под запись flood в PostgreSQL.

Репозиторий resource usage считает расход per-op за последний час, читает готовые VIEW для доступного процента ресурса и умеет найти «худший» op на аккаунте — тот, который скорее всего упрётся в лимит первым.

Всё это не висит в воздухе. Около восьмидесяти pytest-кейсов гоняются на живой PostgreSQL — не на моках. Отдельный файл интеграционных тестов проверяет, что блоки B1–B3–B6–B8 работают вместе: enqueue подтягивает настройки из seed, dedup и claim проходят полный lifecycle, rollback транзакции откатывает вставку в очередь.

### Мост между старым discovery и новой очередью

Discovery по-прежнему хранит аккаунты в SQLite, membership clump — в JSON, сессии — файлами на диске. Воркер очереди этого не знает и не должен знать — он смотрит только в PostgreSQL.

Мы написали sync: скрипт собирает union из трёх источников, нормализует имена сессий, маппит admin_blocked в disabled, помечает «есть на диске, но не в clump» как неактивные, а runtime-состояния вроде cooldown и текущей задачи **не затирает** при повторном запуске. Sync идемпотентен — его можно крутить хоть каждый час. На это семнадцать тестов — unit и integration.

### Воркер как отдельный процесс

По ТЗ воркер не живёт внутри FastAPI lifespan — это отдельный asyncio-процесс. Мы его написали: цикл claim → handler → complete или retry, graceful остановка по SIGTERM — текущая задача дорабатывается, новые не берутся. Конфигурация из env: id воркера, интервал опроса, TTL lock, фильтр по типам задач.

В Docker Compose появились два сервиса: один для staging/prod через `.env`, второй — local profile с PostgreSQL, migrate и воркером в одной связке. Stop grace period тридцать секунд, restart unless-stopped — production-привычки с первого дня.

На момент закрытия спринта handler — заглушка. Но release аккаунта в finally, retry при ошибке, логирование — уже на месте. Следующий спринт подменит заглушку настоящим dispatch.

### SessionClump научился слушаться воркера

До спринта clump сам выбирал, на какую сессию повесить канал. Воркер же должен сказать: «канал вот на *этот* аккаунт, потому что я его уже зарезервировал в PG».

Мы добавили `add_channel_on_session` — канал вешается строго на указанную сессию, listener стартует, assignments обновляется. Если канал уже на другой сессии — ошибка, а не тихий перехват. И `move_channel` — перенос одного канала между двумя заданными сессиями, с очисткой source и идемпотентностью при повторном вызове.

Двенадцать тестов SessionClump проходят, восемь из них — новые для этих методов. Это готовый Telethon-слой: когда появится adapter, ему не придётся лезть внутрь clump.

### Безопасный rollout

Feature flags в `.env.example` зафиксированы с дефолтами, которые **не ломают prod**: PG-очередь в API выключена, granular RPH — упрощённый seed, idle-rebalance в clump пока включён. Переключение на новую модель будет явным жестом, не сюрпризом при деплое.

### Что это даёт по факту

За неделю мы получили не слайды и не «скоро будет», а **работающую платформу очереди**. Задачи создаются и dedup'ятся. Воркеры забирают их без гонок. Аккаунты резервируются атомарно. Лимиты считаются per-op. Discovery и PG синхронизируются. Docker поднимает воркер одной командой. Шесть критериев приёмки из раздела 30 ТЗ уже закрыты, ещё три — наполовину: репозитории готовы, dispatch — нет.

Примерно сорок процентов инфраструктуры MVP готово. Ноль процентов бизнес-сценария «клиент добавил канал через PG-очередь и канал реально слушается» — потому что это как раз текущий спринт.

## Что мы делаем сейчас

С восемнадцатого июня идёт объединённая итерация: закрыть хвост второго спринта и пройти третий — MVP add-channels через PostgreSQL. Если прошлый этап был фундаментом, то сейчас мы **проводим коммуникации и включаем свет**.

### Умный воркер

Первое, что нужно доделать — полный цикл dispatch. Сейчас воркер берёт задачу и сразу «выполняет» пустышкой. Должно стать иначе: после claim — загрузить тип задачи, зарезервировать аккаунт (или два для move), проверить, хватает ли ресурса по каждой операции из состава задачи, вызвать adapter, завершить или отложить, в любом случае освободить аккаунт.

Отдельно допишем postpone — отложить задачу на пять минут **без** списания лимита и **без** increment attempt_count. Это ключевое отличие от ошибки: «сейчас нет свободного аккаунта» не должно стопорить всю очередь. Воркер откладывает одну задачу и берёт следующую.

Для move-channel — dual reserve: source и target блокируются в одной транзакции, иначе можно зависнуть с каналом между двумя мирами. Watchdog будет помечать зависшие in_progress как stuck по таймауту из типа задачи — чтобы задачи не висели вечно после падения процесса.

Сначала adapter будет mock — без Telethon, только фиксация факта вызова. Так мы закроем интеграционные тесты и multi-worker сценарий с двумя процессами, не дожидаясь реального Telegram. Потом mock заменится на Telethon без переписывания dispatch — это осознанное решение, а не компромисс.

### MVP: add-channels через PG

Когда dispatch живой, подключаем настоящий adapter: задача `parser_add_channel` превращается в вызов `add_channel_on_session` на сессии, которую воркер зарезервировал. Перед Telethon RPC — INSERT в account_resource_usage по каждой операции из состава задачи: расход фиксируется в момент передачи аккаунту, как требует ТЗ, в том числе для неуспешных попыток.

В API появится переключатель USE_PG_QUEUE: bulk add-channels раскладывается в N атомарных задач в PostgreSQL вместо SQLite bulk. Флаг по умолчанию false — legacy не ломается. Dual-write обновит assigned_account_id и в PG, и в JSON clump — на переходный период два источника должны сходиться.

Flood и ban из HealthMonitor начнут попадать в PostgreSQL: cooldown переживает рестарт воркера, pick аккаунта это учитывает. Клиент сможет спросить статус задачи через API — status, attempt_count, last_error.

Финальная точка спринта — E2E на staging: реальный POST add-channels с флагом, реальная задача в PG, реальный воркер, реальный канал слушается на назначенной сессии. Это и есть MVP.

### Что будет готово, когда спринт закончится

Воркер будет работать по полному §5.8 ТЗ — reserve, проверка ресурса, postpone, retry, release. Очередь не будет стопориться, если один аккаунт занят или «на грани» лимита. Add-channels можно будет включить на staging под флагом без отката legacy. n8n и другие клиенты увидят статус async-операции.

По приёмке §30 закроются критерии про подбор аккаунта, postpone, per-op расход, dual-account move, зависшие задачи и главный MVP-сценарий.

Сознательно останется за бортом: remove-channels через PG (remove пока на SQLite), полная история task_attempts (спринт четырёх), typed errors и backoff, автоматические продюсеры ±5%, cleanup legacy action_queue. Это не провалы — это следующие этапы дорожной карты.

## Было и будет — одним абзацем на контраст

**После прошлого спринта** очередь умела хранить и отдавать задачи, но не исполняла их по правилам бизнеса. Воркер крутился, заглушка ставила done, Telethon не вызывался, API всё ещё на SQLite bulk, flood жил только в памяти процесса.

**После текущего** цепочка замкнётся: API → PostgreSQL → воркер → Telethon → clump. Нет аккаунта — отложим, не упадём. Нет ресурса — отложим, не сожжём лимит. Два воркера — без дублей. Flood — в базе, переживёт рестарт. Add-channels — первый production-ready cutover под флагом.

---

## О чём стоит помнить по ходу

Mock-first для dispatch — не откладывание Telethon ради экономии, а способ не блокировать тесты и multi-worker на реальном API Telegram. Remove-channels при включённой PG-очереди останется на старом пути — это нужно явно сказать в runbook, иначе половина API будет «новой», половина «старой». Если сроки подожмут, MVP add-channel можно выпустить без move-adapter и без полного flood-sync — они сдвинутся на несколько дней, ядро не развалится.

## В конце

Прошлый спринт дал нам **production-grade фундамент**: схему с мониторингом, seed из реального discovery, typed репозитории, девяносто с лишним автотестов, sync аккаунтов, Docker-воркер и SessionClump, готовый слушаться очереди. Это уже можно показывать, демонстрировать, review'ить — не «мы думаем сделать очередь», а «очередь есть, вот pytest green».

Текущий спринт превращает фундамент в **продукт**: dispatch по правилам ТЗ и первый настоящий сценарий add-channels через PostgreSQL на staging. После него можно включать флаг, собирать обратную связь и не ждать продюсеров, полного retry-контура и вычищения legacy — они придут в следующих спринтах, но платформа уже будет стоять на ногах.
