5. **Модель лимитов: per-op RPH вместо `hourly_limit` (решение по схеме БД).** Исходное ТЗ (§6–§8, §11) описывает единый `accounts.hourly_limit` и поля `resource_cost` / `source_resource_cost` / `target_resource_cost` в `task_types`. Утверждённая схема (`DB/BD_schema.sql`, `queue-balancer-schema.dbml`) **заменяет** это на **per-op учёт Telethon-операций**:

   | Было (исходное ТЗ) | Стало (схема PG) |
   |--------------------|------------------|
   | `accounts.hourly_limit` | `resource_op_types.rph_limit` + `reserve_percent` |
   | `task_types.resource_cost*` | `task_type_ops` (связь task ↔ op, `units_per_execution`, `account_role`) |
   | `account_resource_usage.resource_cost` | `account_resource_usage.op_type_id` + `units` |
   | `available = hourly_limit − used` | `available_op = effective_rph − count за час по op_type_id`; effective_rph = rph_limit × (1 − reserve_percent/100) |

   **Почему:** лимиты Telegram привязаны к типам RPC (`JoinChannel`, `get_entity`, `iter_messages`…), а не к абстрактной «обработке»; стыкуется с `queue_prot_blance/ops_catalog.py` и multi-op пайплайнами (`collect_extra_data`).

   **Семантика §7 ТЗ сохраняется:** расход фиксируется при передаче задачи аккаунту; неуспешные попытки учитываются; postpone не расходует ресурс; `move_channel` списывает по обоим аккаунтам через `account_role` (`source` / `target`).

   **Карта влияния на задачи плана** (ниже по документу карточки обновлены; пометка **per-op**):

   | Задача | Что меняется |
   |--------|--------------|
   | **A3** | В `task_types` **нет** `resource_cost*`; добавлены миграции `resource_op_types`, `task_type_ops` |
   | **A4** | **Нет** `hourly_limit`; лимиты — в справочнике op-типов |
   | **A6** | Запись привязана к `op_type_id`, не к scalar `resource_cost` |
   | **A9** | Seed: `resource_op_types` (`seed-resource-op-types.sql`) + `task_type_ops` для MVP-типов |
   | **A11** | Миграции включают op-таблицы и seed |
   | **B2** | Загрузка `task_type_ops` вместе с типом задачи |
   | **B8** | `count_last_hour(account_id, op_type_id)`; агрегация по всем op задачи |
   | **C5** | Проверка ресурса: **каждый** op из `task_type_ops` ≥ `min_available_resource_percent` |
   | **C4, D4** | Dual-account: op'ы с `account_role=source/target` |

## 1. Контекст и цель

ТЗ требует **платформенную доменную очередь задач в PostgreSQL**, распределяющую работу с Telegram-каналами между аккаунтами так, чтобы:

- не превышать **почасовые лимиты Telethon-операций** на аккаунт (per-op RPH, см. §0.5) и не доводить их до бана;
- равномерно распределять каналы между аккаунтами (±5%);
- выполнять задачи по приоритету и времени создания;
- считать **все реальные попытки** работы через аккаунт (включая неуспешные);
- корректно откладывать задачи, которые сейчас выполнить нельзя, не стопоря очередь;
- отслеживать зависание/рост очереди и системные ошибки.

**Главный принцип ТЗ:** одна строка очереди = одна единица работы с аккаунтом (500 действий → 500 задач).

**Что НЕ меняем (§2 ТЗ):** контур поиска `/discover`, скоринг; таблицу каналов не проектируем с нуля, а расширяем. Исполнение Telethon (`SessionClump`, `Parser_client`, supervisor) переиспользуется как исполнительный слой.

**Принцип внедрения:** минимальный diff в `standalone_discovery` — не переписывать `SessionClump`, а добавить PG-слой (очередь + диспетчер + адаптер) поверх существующего Telethon-исполнения.

## 2. Целевая архитектура (слои)

```
[Источники задач]            [PostgreSQL — источник истины]        [Исполнение — существующий код]
Продюсеры (cron) ─INSERT─►  task_types ◄── настройки dispatch
API (HTTP)       ─INSERT─►  task_type_ops ──► resource_op_types (RPH-лимиты per op)
                            task_queue
                            accounts ◄──── HealthMonitor (cooldown/ban)
                            account_resource_usage (op_type_id + units)
                            task_attempts
                                  │
                                  ▼
                            Worker-Диспетчер (claim → reserve → проверка op-ресурса → postpone/retry)
                                  │
                                  ▼
                            queue/adapter.py (execute_task)
                                  │
                                  ▼
                            SessionClump / Parser_client (Telethon)
```

| Слой | Ответственность | Новый код |
|------|-----------------|-----------|
| Продюсеры / API | Создают строки в `task_queue`, не вызывают Telethon | `scripts/producers/`, правка `parser_router.py` |
| Worker-Диспетчер | Выбор задачи, резерв аккаунта, жизненный цикл статусов | `queue_worker.py`, `queue/repos/` |
| Адаптер | Маппинг `task_type` → вызов clump; ошибки → retry/fail | `queue/adapter.py` (или `queue_prot.py`) |
| SessionClump | Telethon: add, move, listen | +2 точечных метода, hook flood→PG |

**Запрет:** не расширять `session_registry.py` логикой очереди — только точечные методы и sync cooldown.

- **Висячие зависимости** (см. §0.2): `Z1`, `A1` отсутствуют — закрыть как неявные предпосылки S1. **`Z2` закрыт:** `source_channels`.
