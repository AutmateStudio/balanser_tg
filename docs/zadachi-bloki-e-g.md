# Задачи блоков E–G — очередь PostgreSQL (Lidogen Telegram Balancer)

**Дата:** 2026-06-24  
**Источники:** `docs(plan)/итоговый-план-разработки.md`, `docs(plan)/_gen_tasks_xlsx.py`, `docs(plan)/tz-extract.txt` (§10, §20, §21–24, §26–29), `docs(plan)/diff-tz-vs-standalone-discovery.md`

Документ описывает **все запланированные задачи после MVP (S3)** — блоки **E** (ошибки и идемпотентность), **F** (продюсеры), **G** (мониторинг и оповещения), включая **G6** и **G7** из утверждённого ТЗ.

### G7★. Оповещение в чат при достижении порогов загрузки

| | |
|---|---|
| **Спринт** | S5 |
| **Зависимости** | G3, G4 (предложено) |
| **Статус в ТЗ** | Новое требование утверждённого ТЗ; часы не заданы — **требуется уточнение порогов** |
| **Формулировка ТЗ** | «Оповещение в чат при достижении 75% лимита по каналам на все аккаунты, процентов загруженности 100%». |
| **per-op (§0.5)** | «100% загрузки ресурса» = хотя бы один критичный op аккаунта исчерпал effective_rph (или сводный порог по худшему op). 75% по каналам — счёт по `source_channels` / `max_channels_per_session`. |
| **Приёмка (предложено)** | При ≥75% лимита каналов суммарно и/или 100% по ключевому op — сообщение в заданный чат (Telegram/Slack). |
| **Открытые вопросы** | Какой чат; «худший op» vs сводка; debounce (не спамить каждые 5 мин). |

**Возможная декомпозиция G7★ (для оценки):**

| Подзадача | Суть |
|-----------|------|

| G7c | Интеграция с Telegram Bot / webhook |
| G7d | Cooldown между повторными алертами одного типа |
