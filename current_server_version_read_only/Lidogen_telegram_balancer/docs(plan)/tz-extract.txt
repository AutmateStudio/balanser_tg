ТЗ: балансировщик нагрузки для работы с Telegram-каналами через аккаунты
1. Цель задачи
Нужно реализовать балансировщик нагрузки для Telegram-аккаунтов.
Балансировщик должен распределять задачи по работе с каналами между нашими аккаунтами так, чтобы:
не превышать лимиты обработки для каждого аккаунта;
не доводить аккаунты до бана из-за превышения лимитов;
равномерно распределять каналы между аккаунтами;
давать возможность разным инструментам создавать задачи в общую очередь;
выполнять задачи в порядке приоритета;
корректно обрабатывать ситуации, когда задача временно не может быть выполнена;
отслеживать зависание очереди, рост очереди и системные ошибки.
Главный принцип: одна задача в очереди - это одна единица работы с аккаунтом.
Если нужно выполнить много однотипных действий, они должны быть разбиты на отдельные задачи в очереди. Например, если нужно обработать 500 сообщений, это должно быть 500 отдельных задач, а не одна большая задача.
2. Что не меняем
2.1. Контур поиска
Поиск остаётся в текущей архитектуре.
Мы находим каналы и в сыром виде записываем их в БД с теми данными, которые получили на поверхности.
Балансировщик не должен менять текущую логику поиска. Он работает уже с теми каналами, которые есть в БД.
2.2. Существующая таблица каналов
Таблица каналов уже есть.
В рамках этой задачи не нужно проектировать её с нуля. Нужно использовать существующую таблицу каналов и, если потребуется, добавить к ней недостающие поля или сделать связанные таблицы.
Важно: канал может быть закреплён только за одним аккаунтом. Ситуации, когда один и тот же канал одновременно подключён к нескольким аккаунтам, в рамках этой логики быть не должно.
3. Основные сущности
В системе должны быть следующие сущности:
Каналы.
Telegram-аккаунты.
Типы задач.
Очередь задач.
Попытки выполнения задач.
Учёт расхода ресурса аккаунтов.
Инструменты, которые создают задачи в очередь.
Балансировщик, который берёт задачи из очереди и назначает их аккаунтам.
Инструмент мониторинга, который проверяет состояние очереди и аккаунтов.
4. Ключевые термины
4.1. Канал
Канал - это Telegram-канал, который найден контуром поиска и записан в БД.
Канал может быть:
найденным, но ещё не подключённым;
подключённым к одному из аккаунтов;
требующим сбора дополнительных данных;
требующим обновления данных;
требующим переноса на другой аккаунт.
4.2. Аккаунт
Аккаунт - это наш Telegram-аккаунт, через который выполняются действия с каналами.
У каждого аккаунта есть лимит обработок в час. Если лимит превысить, есть риск получить бан или ограничения со стороны Telegram.
Поэтому балансировщик должен считать все попытки работы через аккаунт, включая неуспешные попытки.
4.3. Ресурс аккаунта
Ресурс аккаунта - это доступное количество обработок, которое аккаунт ещё может выполнить в рамках лимита.
Ресурс считается на основе лимита аккаунта и количества уже выполненных попыток за последний час.
Важно: считаются все попытки, а не только успешные задачи.
Если задача была передана аккаунту в работу, такая попытка должна быть учтена в расходе ресурса независимо от результата выполнения.
Если балансировщик только проверил задачу и понял, что аккаунт не подходит, но не запускал работу через аккаунт, это не считается расходом ресурса.
4.4. Задача
Задача - это одна строка в очереди.
Одна задача должна соответствовать одной единице работы с аккаунтом.
Примеры задач:
обновить данные по одному каналу;
собрать одну единицу дополнительных данных;
выполнить одно действие по переносу канала;
выполнить одно действие, созданное другим инструментом.
Задача не должна содержать внутри себя большой пакет действий, который может непредсказуемо расходовать ресурс аккаунта.
4.5. Тип задачи
Тип задачи определяет настройки и ограничения для задач этого типа.
Например:
приоритет;
стоимость по ресурсу;
минимальный свободный ресурс аккаунта;
максимальное число попыток;
размер поддерживаемой очереди;
правила отложенного повторного выполнения;
нужно ли задаче выполнять действие на конкретном аккаунте;
может ли задача выполняться на любом подходящем аккаунте.
5. Общая архитектура
Система должна состоять из следующих частей.
5.1. Таблица каналов
Существующая таблица, где хранятся данные о Telegram-каналах.
В рамках этой задачи таблица используется как источник данных для создания задач и расчётов.
Если в текущей таблице нет нужных полей для работы балансировщика, разработчик должен предложить минимальное расширение схемы.
Критично, чтобы по каналу можно было определить:
к какому аккаунту он сейчас закреплён;
прошёл ли он сбор дополнительных данных;
когда он последний раз обновлялся;
есть ли по нему активная задача в очереди;
можно ли с ним сейчас выполнять действия.
5.2. Таблица аккаунтов
Таблица, где хранятся все наши Telegram-аккаунты и их рабочие параметры.
Через неё балансировщик должен понимать:
какие аккаунты активны;
какие аккаунты временно недоступны;
какие аккаунты находятся в cooldown;
какие аккаунты уже выполняют задачу;
сколько ресурса у аккаунта осталось;
какой лимит у аккаунта в час.
5.3. Таблица типов задач
Таблица, где описаны настройки каждого типа задачи.
Именно здесь должны задаваться лимиты, приоритеты, размер поддерживаемой очереди и другие параметры.
Не нужно хардкодить значения вроде “20 задач в очереди”, “80% свободного ресурса” или “90% свободного ресурса” внутри кода. Эти значения должны быть настройками типа задачи.
5.4. Таблица очереди задач
Таблица, где хранятся задачи, которые нужно выполнить.
Балансировщик берёт задачи из этой таблицы, проверяет возможность выполнения, назначает аккаунт и запускает работу.
5.5. Таблица попыток выполнения задач
Таблица, где фиксируются попытки выполнения задач.
Нужна для диагностики, истории, анализа ошибок и понимания, сколько раз задача реально пыталась выполниться.
5.6. Таблица расхода ресурса аккаунтов
Таблица, где фиксируются все попытки работы через аккаунты.
На основе этой таблицы считается, сколько обработок аккаунт уже сделал за последний час.
Важно: в эту таблицу должны попадать не только успешные выполнения, но и неуспешные попытки, если действие уже было передано аккаунту в работу.
5.7. Инструменты создания задач
Это инструменты, которые анализируют состояние данных и кладут задачи в очередь.
В этом ТЗ описаны несколько таких инструментов:
балансировщик каналов между аккаунтами;
сбор дополнительных данных о каналах;
обновление данных о старых каналах.
Также могут быть другие инструменты с более высоким приоритетом. Они будут описываться отдельно, но должны использовать ту же общую таблицу очереди и таблицу типов задач.
5.8. Балансировщик
Балансировщик - это инструмент, который:
смотрит очередь задач;
выбирает подходящую задачу;
проверяет ограничения типа задачи;
выбирает аккаунт, если он не указан явно;
проверяет ресурс аккаунта;
запускает выполнение задачи;
фиксирует результат выполнения;
если задачу нельзя выполнить сейчас, откладывает её и переходит к следующей.
5.9. Мониторинг
Мониторинг - это отдельная обвязка, которая проверяет, что очередь и балансировщик работают штатно.
Он должен отслеживать:
не зависла ли очередь;
берутся ли задачи в работу;
не растёт ли очередь бесконечно;
нет ли слишком большого числа отложенных задач;
нет ли задач, которые слишком долго находятся в работе;
не закончился ли ресурс у всех аккаунтов;
нет ли массовых ошибок по аккаунтам или типам задач.

7. Расчёт ресурса аккаунта
7.1. Базовое правило
Ресурс аккаунта считается по количеству попыток работы через аккаунт за последний час.
Пример:
лимит аккаунта: 100 обработок в час;
за последний час было зафиксировано 35 попыток;
доступный ресурс: 65;
свободный ресурс в процентах: 65%.
Формула:
available_resource = hourly_limit - used_attempts_last_houravailable_resource_percent = available_resource / hourly_limit * 100
7.2. Что считается расходом ресурса
Расходом ресурса считается любая попытка выполнения задачи через аккаунт.
Сюда входят:
успешные выполнения;
неуспешные выполнения;
попытки, которые завершились ошибкой;
попытки, которые были прерваны после старта выполнения.
Не считаются расходом ресурса:
просмотр очереди балансировщиком;
проверка доступности задачи;
проверка доступности аккаунта;
ситуация, когда задача была отложена до передачи аккаунту в работу.
7.3. Когда фиксировать расход ресурса
Расход ресурса нужно фиксировать в момент, когда задача фактически передаётся аккаунту в работу.
Это важно, потому что даже неуспешная попытка может повлиять на лимиты Telegram.
7.4. Стоимость задачи по ресурсу
По умолчанию одна задача стоит 1 единицу ресурса.
Но стоимость должна задаваться в типе задачи.
Например:
Тип задачи
Стоимость
update_channel
1
collect_extra_data
1
move_channel
1 для исходного аккаунта и 1 для целевого аккаунта
Если задача использует два аккаунта, например при переносе канала, ресурс должен учитываться по обоим аккаунтам.

8.2. Приоритеты
Приоритет должен быть числовым.
Чем больше число, тем выше приоритет.
Пример:
Приоритет
Значение
1000
Срочные системные или пользовательские задачи
500
Важные регулярные задачи
200
Сбор дополнительных данных
100
Балансировка каналов
50
Обновление старых каналов
Конкретные значения приоритетов можно изменить, но принцип должен быть единым: большее значение означает более высокий приоритет.
8.3. Размер поддерживаемой очереди
Число задач, которое инструмент должен держать в очереди, должно задаваться в поле target_queue_size.
Например, если для типа задачи указано target_queue_size = 20, инструмент должен следить, чтобы в очереди было до 20 активных задач этого типа.
Под активными задачами здесь понимаются задачи в статусах:
queued;
scheduled;
retry;
in_progress.
Завершённые и отменённые задачи не должны учитываться в этом лимите.

9.3. attempt_count и postpone_count
Важно разделять два разных счётчика.
attempt_count - это количество реальных попыток выполнения задачи через аккаунт.
Этот счётчик увеличивается только тогда, когда задача была передана аккаунту в работу.
postpone_count - это количество случаев, когда балансировщик пытался взять задачу, но не смог её выполнить сейчас и отложил на будущее.
Например:
нет доступного ресурса у конкретного аккаунта;
аккаунт в cooldown;
нет свободных аккаунтов для задачи без конкретного аккаунта;
задача временно не может быть выполнена по ограничениям типа задачи.
postpone_count не должен расходовать ресурс аккаунта, но должен использоваться для мониторинга.
Если задачу слишком много раз откладывали, система должна сигнализировать о проблеме.

Это особенно важно для задач переноса канала между аккаунтами.
12. Защита от дублей задач
Инструменты, которые создают задачи, не должны бесконечно создавать одинаковые задачи.
Для этого у задач должен быть dedup_key.
Примеры:
update_channel:{channel_id}collect_extra_data:{channel_id}move_channel:{channel_id}:{source_account_id}:{target_account_id}
Перед созданием задачи инструмент должен проверить, нет ли уже активной задачи с таким же dedup_key.
Активными считаются задачи в статусах:
queued;
scheduled;
retry;
in_progress.
Если активная задача с таким dedup_key уже есть, новую задачу создавать не нужно.
13. Жизненный цикл задачи
13.1. Обычный успешный сценарий
Инструмент создаёт задачу в статусе queued.
Балансировщик находит задачу.
Балансировщик проверяет тип задачи.
Балансировщик проверяет доступность аккаунта или выбирает подходящий аккаунт.
Балансировщик переводит задачу в in_progress.
Балансировщик фиксирует попытку выполнения.
Балансировщик фиксирует расход ресурса аккаунта.
Задача выполняется.
При успехе задача переводится в done.
13.2. Если задачу нельзя выполнить сейчас
Если задачу нельзя выполнить сейчас, но это не финальная ошибка, балансировщик должен:
увеличить postpone_count;
записать причину в last_error;
поставить новое время run_after;
оставить задачу в статусе scheduled или retry;
перейти к следующей задаче.
Примеры причин:
у конкретного аккаунта недостаточно ресурса;
аккаунт временно недоступен;
аккаунт находится в cooldown;
нет подходящих свободных аккаунтов;
ограничения типа задачи не позволяют запустить задачу сейчас.
13.3. Если задача была запущена, но завершилась ошибкой
Если задача уже была передана аккаунту в работу, это считается попыткой.
В этом случае нужно:
увеличить attempt_count;
записать попытку в task_attempts;
записать расход ресурса в account_resource_usage;
записать ошибку;
понять, можно ли повторить задачу.
Если attempt_count < max_attempts, задачу можно перевести в retry и назначить новое run_after.
Если attempt_count >= max_attempts, задачу нужно перевести в failed.
13.4. Если задача зависла
Если задача находится в in_progress дольше допустимого времени, её нужно считать зависшей.
Допустимое время берётся из типа задачи: task_timeout_seconds.
Если задача зависла, система должна:
перевести её в stuck или вернуть в retry;
записать причину;
увеличить соответствующий счётчик;
при необходимости отправить сигнал в мониторинг.
Конкретный механизм восстановления зависших задач может быть выбран разработчиком, но задача не должна бесконечно оставаться в in_progress.
14. Атомарное взятие задачи в работу
Балансировщик должен брать задачу в работу атомарно.
Нельзя допустить ситуацию, когда два процесса одновременно взяли одну и ту же задачу.
При взятии задачи нужно:
выбрать подходящую задачу;
заблокировать её для других процессов;
проставить locked_by;
проставить locked_at;
проставить locked_until;
перевести задачу в in_progress;
назначить аккаунт, если он не был указан.
Если в проекте используется PostgreSQL, можно реализовать это через транзакцию и блокировку строк.
Конкретную реализацию выбирает разработчик, но требование обязательное: одна задача не должна одновременно выполняться двумя воркерами.
15. Алгоритм выбора задачи
Балансировщик должен выбирать задачи по следующим правилам.
15.1. Базовая фильтрация
Балансировщик рассматривает только задачи, которые:
находятся в статусе queued или retry;
имеют run_after <= now();
относятся к включённому типу задачи;
не заблокированы другим процессом;
не превышают max_attempts.
15.2. Сортировка
Задачи сортируются:
по приоритету - от большего к меньшему;
по времени создания - от более старой к более новой.
То есть сначала берутся более приоритетные задачи, а внутри одного приоритета - более старые.
15.3. Если первая задача не может быть выполнена
Если задача подходит по очереди, но не может быть выполнена прямо сейчас, балансировщик не должен останавливать всю очередь.
Он должен:
отложить эту задачу;
зафиксировать причину;
увеличить postpone_count;
перейти к следующей подходящей задаче.
Это нужно, чтобы одна проблемная задача не блокировала всю очередь.
16. Алгоритм выбора аккаунта
16.1. Если у задачи указан конкретный аккаунт
Если в задаче указан account_id, задача должна выполняться именно на этом аккаунте.
Балансировщик проверяет:
аккаунт существует;
аккаунт включён;
аккаунт имеет статус active;
аккаунт не находится в cooldown;
аккаунт сейчас не выполняет другую задачу;
у аккаунта достаточно ресурса;
свободный ресурс аккаунта соответствует ограничениям типа задачи.
Если все условия выполнены, задача запускается на этом аккаунте.
Если условия не выполнены, задача откладывается. Балансировщик увеличивает postpone_count, ставит новое run_after и переходит к следующей задаче.
16.2. Если у задачи нет конкретного аккаунта
Если у задачи нет конкретного аккаунта, балансировщик должен выбрать подходящий аккаунт сам.
Он должен искать аккаунт среди доступных аккаунтов.
Аккаунт подходит, если:
он включён;
он в статусе active;
он не находится в cooldown;
он не выполняет другую задачу;
у него достаточно ресурса для задачи;
его свободный ресурс соответствует ограничениям типа задачи.
Если один аккаунт не подходит, балансировщик должен выбрать другой.
Задачу нужно откладывать только если среди аккаунтов не найдено ни одного подходящего.
16.3. Критерий выбора самого подходящего аккаунта
Если доступно несколько аккаунтов, нужно выбрать самый ненагруженный.
В базовой версии самый ненагруженный аккаунт - это аккаунт, который:
сейчас не выполняет задач;
имеет достаточный свободный ресурс;
за последний час выполнил меньше всего задач;
имеет наиболее высокий процент свободного ресурса.
Конкретную формулу выбора можно оставить разработчику, но она должна учитывать лимит аккаунта и не должна выбирать аккаунт, который находится на грани лимита.
17. Ограничение параллельности
На первом этапе один аккаунт должен выполнять не более одной задачи одновременно.
Это снижает риск превышения лимитов и упрощает контроль ресурса.
Если в будущем понадобится параллельное выполнение нескольких задач одним аккаунтом, это должно быть отдельной доработкой с отдельными лимитами и правилами.
18. Обработка задач с двумя аккаунтами
Некоторые задачи могут использовать два аккаунта.
Главный пример - перенос канала с одного аккаунта на другой.
В этом случае задача должна содержать:
source_account_id;
target_account_id;
channel_id.
Перед запуском нужно проверить ресурс обоих аккаунтов.
Если ресурса не хватает хотя бы у одного аккаунта, задачу нельзя запускать.
Если задача запускается, расход ресурса должен быть учтён по обоим аккаунтам.
Стоимость по каждому аккаунту должна задаваться в типе задачи:
source_resource_cost;
target_resource_cost.
19. Перенос канала между аккаунтами
19.1. Общая логика
Перенос канала означает, что канал должен быть убран с одного аккаунта и закреплён за другим.
Канал не может одновременно считаться подключённым к нескольким аккаунтам.
Задача переноса должна быть доведена до конечного согласованного состояния.
Нельзя допустить ситуацию, когда:
канал потерял связь со старым аккаунтом, но не был закреплён за новым;
канал закреплён в БД за новым аккаунтом, но фактически перенос не выполнен;
задача считается успешной, хотя часть действий не завершилась;
при повторе задачи система выполняет опасные дублирующие действия.
19.2. Что обязательно нужно учесть
При переносе нужно учитывать, что операция расходует ресурс обоих аккаунтов:
исходного аккаунта;
целевого аккаунта.
Также задача переноса должна быть безопасной при повторном запуске.
Если задача упала на середине, повторный запуск должен понимать текущее состояние и продолжить или корректно завершить процесс.
19.3. Что остаётся на разработчика
Конкретный технический сценарий переноса остаётся на разработчика.
Разработчик должен сам определить, как лучше реализовать процесс:
через несколько внутренних этапов одной задачи;
через несколько связанных задач;
через отдельное поле стадии переноса;
через компенсационную операцию;
через ручной флаг для спорных ситуаций.
Но обязательные требования такие:
перенос не должен оставлять канал в неопределённом состоянии;
задача не должна считаться done, пока перенос не завершён полностью;
расход ресурса должен учитываться по обоим аккаунтам;
при сбое должна быть возможность понять, на каком этапе остановился перенос;
повтор задачи не должен ломать данные.
20. Retry-логика
20.1. Почему нужен retry
Некоторые ошибки могут быть временными.
Например:
аккаунт временно недоступен;
не хватило ресурса;
Telegram вернул временное ограничение;
произошла сетевая ошибка;
воркер упал во время выполнения.
Такие задачи не нужно сразу считать окончательно проваленными.
20.2. Настройка retry
Параметры retry должны задаваться в типе задачи:
max_attempts;
retry_delay_seconds;
retry_backoff_multiplier;
max_retry_delay_seconds.
Пример:
1-я ошибка: повтор через 10 секунд2-я ошибка: повтор через 20 секунд3-я ошибка: повтор через 40 секунд4-я ошибка: повтор через 80 секунд
При этом задержка не должна превышать max_retry_delay_seconds.
20.3. Нехватка ресурса - не всегда ошибка
Если задача не может выполниться из-за отсутствия ресурса аккаунта, это может быть нормальной ситуацией.
В этом случае задача должна быть отложена через run_after.
Но если задача откладывается слишком много раз, это уже повод для мониторинга.
Для этого используется postpone_count.
21. Инструменты, создающие задачи
21.1. Общие правила для всех инструментов
Любой инструмент, который создаёт задачи, должен:
работать через таблицу task_queue;
использовать настройки из task_types;
не создавать дубли активных задач;
использовать dedup_key;
учитывать target_queue_size;
не хардкодить лимиты и размеры очереди;
создавать задачи с корректным приоритетом.
22. Инструмент: балансировщик каналов
22.1. Назначение
Инструмент следит за тем, чтобы между аккаунтами было распределено примерно одинаковое количество каналов.
Допустимое отклонение: ±5%.
На этом этапе балансировка выполняется именно по количеству каналов. Дополнительные веса каналов учитывать не нужно.
22.2. Логика работы
Инструмент:
считает количество каналов, закреплённых за каждым активным аккаунтом;
определяет аккаунты, где каналов слишком много;
определяет аккаунты, где каналов слишком мало;
создаёт задачи на перенос каналов между аккаунтами;
поддерживает в очереди количество задач, указанное в target_queue_size для этого типа задачи.
Например, если в типе задачи указано target_queue_size = 20, инструмент должен держать до 20 активных задач на выравнивание баланса.
22.3. Ограничение по ресурсу
Инструмент должен создавать или запускать такие задачи только с учётом настроек типа задачи.
Например, если в типе задачи указано, что задача может выполняться только при свободном ресурсе более 80%, это должно проверяться через min_available_resource_percent.
22.4. Приоритет
Задачи балансировки каналов должны иметь низкий приоритет.
Они не должны мешать более важным задачам.
23. Инструмент: сбор дополнительных данных о каналах
23.1. Назначение
Инструмент ищет каналы, которые уже подключены, но по которым ещё не были собраны дополнительные данные.
Под дополнительными данными понимаются:
сообщения;
детализация по каналу;
другие данные, которые нужны продукту.
23.2. Логика работы
Инструмент:
ищет пул каналов, по которым не собраны дополнительные данные;
проверяет, нет ли уже активной задачи на этот канал;
создаёт задачи в очередь;
поддерживает количество активных задач по настройке target_queue_size.
Например, если для типа задачи указано target_queue_size = 20, инструмент держит до 20 активных задач на сбор дополнительных данных.
23.3. Ограничение по ресурсу
Ограничения по ресурсу должны задаваться в типе задачи.
Например, задача может запускаться только если у подходящих аккаунтов свободно более 90% ресурса.
23.4. Приоритет
У этого типа задач один из самых низких приоритетов.
Он должен выполняться только тогда, когда у аккаунтов есть свободный ресурс.
24. Инструмент: обновление данных о канале
24.1. Назначение
Инструмент нужен для того, чтобы в свободное время обновлять данные по старым каналам.
24.2. Логика выбора каналов
Критерии того, какие каналы считать старыми и требующими обновления, в этом ТЗ подробно не описываются.
Разработчик должен использовать существующую таблицу каналов и текущую продуктовую логику актуальности данных.
24.3. Очередь
Инструмент должен поддерживать количество активных задач, указанное в target_queue_size для соответствующего типа задачи.
Например, если указано 20, инструмент держит до 20 активных задач.
24.4. Ограничение по ресурсу
Ограничения по ресурсу задаются в типе задачи.
Например, задача может запускаться только если у подходящих аккаунтов свободно более 90% ресурса.
24.5. Приоритет
Этот инструмент имеет самый низкий приоритет.
Он не должен мешать более важным задачам.
25. Другие инструменты
В системе могут быть другие инструменты, которые будут создавать задачи с более высоким приоритетом.
Они не описываются в этом ТЗ подробно.
Но они должны работать по тем же правилам:
создавать задачи в task_queue;
использовать task_types;
не создавать дубли;
учитывать лимиты аккаунтов;
корректно задавать приоритет;
не обходить балансировщик.
26. Мониторинг работы балансировщика
Для балансировщика нужно сделать отдельную обвязку мониторинга.
Её задача - смотреть на показатели в БД и сигнализировать, если система работает нештатно.
26.1. Что нужно проверять
Мониторинг должен проверять:
очередь не зависла;
задачи штатно берутся в работу;
очередь не растёт бесконечно;
задачи не находятся слишком долго в in_progress;
нет задач, которые слишком много раз откладываются;
нет массовых ошибок по одному типу задач;
нет массовых ошибок по одному аккаунту;
аккаунты не превышают лимиты;
нет ситуации, когда все аккаунты недоступны;
нет ситуации, когда очередь есть, но выполнений давно нет.
26.2. Метрики очереди
Желательно считать следующие показатели:
Метрика
Что показывает
queue_size_total
Общее количество активных задач
queue_size_by_status
Количество задач по статусам
queue_size_by_type
Количество задач по типам
oldest_queued_task_age
Возраст самой старой задачи в очереди
in_progress_count
Количество задач в работе
stuck_tasks_count
Количество зависших задач
failed_tasks_count
Количество проваленных задач
retry_tasks_count
Количество задач на повторе
postponed_tasks_count
Количество задач, которые часто откладываются
done_tasks_last_5_min
Сколько задач выполнено за последние 5 минут
26.3. Метрики аккаунтов
Желательно считать следующие показатели:
Метрика
Что показывает
account_used_last_hour
Сколько ресурса аккаунт использовал за последний час
account_available_resource
Сколько ресурса осталось
account_available_resource_percent
Процент свободного ресурса
account_status
Текущий статус аккаунта
account_error_rate
Частота ошибок аккаунта
accounts_in_cooldown
Сколько аккаунтов в cooldown
active_accounts_count
Сколько аккаунтов реально доступно
accounts_without_resource
Сколько аккаунтов без свободного ресурса
26.4. Когда сигнализировать о проблеме
Система должна сигнализировать, если:
очередь растёт длительное время и не уменьшается;
есть задачи, которые слишком долго находятся в in_progress;
задача была отложена слишком много раз;
один и тот же тип задач массово падает;
один и тот же аккаунт массово даёт ошибки;
все аккаунты находятся без ресурса или в cooldown;
есть активная очередь, но задачи давно не завершаются;
старые задачи долго не доходят до выполнения;
какой-либо аккаунт превысил или почти превысил лимит.
Важно: нехватка ресурса сама по себе может быть нормальной ситуацией. Это не всегда ошибка.
Ошибка - это когда такая ситуация длится слишком долго, блокирует очередь или приводит к бесконечному росту задач.
26.5. Что делать при проблеме
Конкретный способ уведомления остаётся на разработчика.
Это может быть:
лог;
алерт;
сообщение в Telegram;
запись в отдельную таблицу системных событий;
уведомление в админке.
Но проблема должна быть видимой, а не просто оставаться в БД.
27. Обработка ошибок
27.1. Виды ошибок
Ошибки желательно разделять на несколько типов.
Тип ошибки
Что означает
Что делать
Временная ошибка
Можно попробовать позже
Перевести задачу в retry
Ошибка ресурса
Сейчас не хватает ресурса
Отложить задачу через run_after
Ошибка аккаунта
Аккаунт не может выполнить задачу
Отправить аккаунт в cooldown или error
Постоянная ошибка
Повтор не поможет
Перевести задачу в failed
Ошибка данных
Некорректные данные в задаче
Перевести задачу в failed или отправить на ручную проверку
27.2. Что обязательно логировать
При ошибке нужно сохранять:
ID задачи;
тип задачи;
аккаунт;
канал;
код ошибки;
текст ошибки;
время ошибки;
номер попытки;
будет ли повтор;
новое время run_after, если задача отложена.
28. Минимальные требования к реализации
Разработчик должен реализовать:
таблицу типов задач;
таблицу очереди задач;
учёт попыток выполнения;
учёт расхода ресурса аккаунтов;
атомарное взятие задачи в работу;
выбор аккаунта с учётом лимитов;
поддержку задач с конкретным аккаунтом;
поддержку задач без конкретного аккаунта;
поддержку задач с двумя аккаунтами;
защиту от дублей через dedup_key;
retry-логику;
отложенное выполнение через run_after;
счётчик реальных попыток attempt_count;
счётчик отложенных запусков postpone_count;
мониторинг зависаний и роста очереди;
инструменты для создания низкоприоритетных задач:
балансировка каналов;
сбор дополнительных данных;
обновление старых каналов.
29. Что можно оставить на усмотрение разработчика
Следующие детали можно оставить разработчику, если он соблюдает описанные выше требования:
точная SQL-схема таблиц;
названия статусов, если в проекте уже есть свои стандарты;
конкретный способ атомарной блокировки задач;
точная формула выбора самого ненагруженного аккаунта;
конкретный сценарий переноса канала между аккаунтами;
способ восстановления зависших задач;
способ отправки алертов;
способ хранения дополнительного payload;
техническая реализация воркеров.
Но разработчик не должен менять основные правила:
не превышать лимиты аккаунтов;
считать все реальные попытки;
не выполнять одну задачу двумя воркерами;
не создавать дубли задач;
не оставлять задачи навсегда зависшими;
не оставлять канал в неопределённом состоянии при переносе;
не подключать один канал к нескольким аккаунтам.
30. Критерии приёмки
Задачу можно считать выполненной, если:
задачи создаются в очередь;
задачи имеют тип, приоритет и настройки из task_types;
балансировщик берёт задачи по приоритету и времени создания;
задачи с run_after в будущем не берутся в работу;
одна задача не может быть взята двумя воркерами одновременно;
задача без конкретного аккаунта получает подходящий свободный аккаунт;
если один аккаунт не подходит, балансировщик ищет другой;
задача с конкретным аккаунтом выполняется только на нём;
если конкретный аккаунт недоступен, задача откладывается;
реальные попытки выполнения учитываются в лимите аккаунта;
неуспешные попытки тоже учитываются в лимите аккаунта;
задачи не запускаются, если у аккаунта недостаточно ресурса;
задачи с двумя аккаунтами проверяют ресурс обоих аккаунтов;
перенос канала учитывает ресурс обоих аккаунтов;
фоновые инструменты не создают дубли активных задач;
количество поддерживаемых задач берётся из target_queue_size;
лимиты запуска берутся из типа задачи;
retry работает через run_after;
задачи не висят бесконечно в in_progress;
есть мониторинг, который сигнализирует о зависании очереди, росте очереди и слишком частых откладываниях задач.
