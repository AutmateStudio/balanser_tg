## 0. Контекст — 1 минута

### О чём проект

Мы переводим балансировщик Telegram-аккаунтов с legacy-механизма (`action_queue` в SQLite, bulk-операции) на **доменную очередь в PostgreSQL**:

- атомарные задачи по каналам;
- приоритеты и dedup;
- per-op лимиты RPH (запросов в час по типам Telethon-операций);
- несколько воркеров без гонок (`SKIP LOCKED`).

### Где мы в дорожной карте

| Спринт | Календарь | Статус |
|--------|-----------|--------|
| **S1** | 2026-06-09 — 2026-06-16 | ✅ Завершён |
| **S2** (начало) | 2026-06-16 — 2026-06-23 | 🟡 Частично (C1, Z4, D1, D2) |
| **S2→S3** (текущий) | 2026-06-18 — ~2026-07-03 | 🔵 В работе |
| S4 | 2026-07-03 — 2026-07-09 | ⬜ Retry, attempts |
| S5 | 2026-07-09 — 2026-07-21 | ⬜ Продюсеры, мониторинг |

### Главная метафора

- **Прошлый спринт (S1)** — фундамент и каркас здания.
- **Текущий спринт (S2→S3)** — провести коммуникации и включить свет: задача реально проходит путь от API до Telethon.

Это ~**40% инфраструктуры MVP** и **нулевая точка** для legacy `action_queue`.

### 1.2. Архитектурный сдвиг: от bulk к платформенной очереди

#### Было (legacy discovery)

```
POST /add-channels  →  action_queue (SQLite)  →  bulk FIFO в SessionClump
                         ↑ один поток, нет dedup, нет per-channel retry
```

#### Стало (после спринта)

```
                    ┌─────────────────────────────────────┐
                    │         PostgreSQL (Shared DB)       │
                    │  task_types · task_queue · accounts  │
                    │  resource_op_types · usage · views   │
                    └──────────────┬──────────────────────┘
                                   │
         enqueue (B3)              │ claim_next SKIP LOCKED (B4)
              ↑                    ↓
    API / продюсеры          queue-worker (C1, Z4)
                                   │
                                   ↓
                         SessionClump (D1, D2) — готов к подключению
```

**Ключевое архитектурное решение — per-op модель лимитов (§0.5 итогового плана):**

- В исходном ТЗ лимит был scalar (`hourly_limit` на аккаунте).
- Мы реализовали **каталог `resource_op_types`**: каждая Telethon-операция (`get_entity`, `channels.JoinChannel`, `iter_messages`…) имеет свой `rph_limit` и `reserve_percent`.
- Задача (`task_type_ops`) описывает **какие op и в каком количестве** она расходует, с ролями `primary` / `source` / `target` для dual-account сценариев.
- Формула доступного ресурса:  
  `effective_rph = rph_limit × (1 − reserve_percent/100)`  
  `available_percent = (effective_rph − used_last_hour) / effective_rph × 100`

Это позволит воркеру (C5) блокировать задачу, если **любой** op из состава задачи «на грани» — точнее, чем один общий счётчик.

### 1.4. Слой доступа к данным — Python-репозитории (блок B)

Построен **типизированный async-слой** на `asyncpg` — единая точка для воркера и будущего FastAPI.

#### B1 — `queue/db.py`: инфраструктура подключения

- Пул: `min_size=1`, `max_size=20`, timeout 20s
- Контекст-менеджеры `acquire()` и `transaction()` с автоматическим commit/rollback
- `healthcheck()` — `SELECT 1` для readiness probe
- `verify_transaction_rollback()` — smoke-тест инварианта транзакций в CI

#### B2 — `per_op_reading.py`: конфигурация задач из БД

- `TaskTypesRepo.get_by_code()` — тип задачи **вместе с op-составом** (JOIN `task_type_ops` + `resource_op_types`)
- Dataclass-модели: `TaskType`, `TaskTypeOp` с `account_role`, `units_per_execution`, `rph_limit`, `reserve_percent`
- `list_enabled()` — только активные типы для dispatch
- Все retry/postpone/timeout параметры читаются из `task_types`, не хардкодятся

#### B3 — `task_queue.py`: постановка в очередь

- `EnqueueInput` / `EnqueueResult` — typed API
- Автоподстановка `priority` и `max_attempts` из `task_types`
- **Dedup:** partial unique index — повторный enqueue с тем же `dedup_key` → `created=False`, `existing_task_id`
- Обработка `UnknownTaskTypeError` для выключенных типов

#### B4 + C7 — `task_queue.py`: атомарный claim

**SQL-алгоритм (production-grade):**

```sql
-- 1) CTE max_prio: максимальный priority среди готовых
-- 2) CTE ready: задачи этого уровня, ORDER BY random() LIMIT 1
-- 3) FOR UPDATE SKIP LOCKED — два воркера не возьмут одну строку
-- 4) UPDATE → in_progress, lock TTL, attempt_count++
```

**Инварианты, покрытые тестами:**

- Два параллельных `claim_next` → разные `id`
- Задача с будущим `run_after` не берётся
- HIGH priority забирается раньше LOW
- При равном priority — случайный выбор (решение §0.1)
- Истёкший lock (`locked_until`) — задача снова доступна
- `complete()` → `done`; `reschedule_or_fail()` → `retry` или `failed` по `max_attempts`

#### B6 — `accounts.py`: управление аккаунтами

- `pick()` — LRU-подобный порядок: `ORDER BY last_used_at ASC NULLS FIRST`
- `reserve()` / `release()` — атомарный `UPDATE … WHERE current_task_id IS NULL`
- **`pick_and_reserve()`** — pick + reserve в **одной транзакции** с `FOR UPDATE SKIP LOCKED` — два воркера не получат один аккаунт
- Учёт `cooldown_until` в SQL pick (готовность к B7)

#### B8 — `resource_usage.py`: per-op учёт RPH

- `insert(account_id, op_type_id, task_id, units)` — запись расхода
- `count_last_hour(account_id, op_type_id)` — сумма `units` за скользящий час
- `op_availability()` — читает VIEW `v_account_op_usage_last_hour`
- `worst_available_percent()` — агрегатор для C5: минимальный % среди всех op аккаунта

### 1.5. Мост discovery ↔ PostgreSQL (A10)

**Проблема:** discovery живёт на SQLite (`telegram_accounts.db`) + JSON (`parser_jobs.json`) + файлы `*.session`. PG-воркер должен видеть **ту же картину аккаунтов**.

**Решение:** `app_balance/queue/accounts_sync.py` + CLI `scripts/sync_accounts_to_pg.py`

| Источник | Что извлекаем |
|----------|---------------|
| SQLite `telegram_accounts` | `session_name`, `admin_blocked` |
| Scan `SESSIONS_DIR` | Аккаунты, существующие на диске |
| `parser_jobs.json` | Membership в clump (активные сессии) |

**Правила маппинга (бизнес-логика):**

| Условие | status | is_enabled | pickable |
|---------|--------|------------|----------|
| `admin_blocked=1` | disabled | false | ❌ |
| На диске, не в clump | disabled | false | ❌ |
| В clump, не blocked | active | true | ✅ |
| Runtime: cooldown/banned/error | **сохраняется** при upsert | — | — |

**Идемпотентность:** повторный запуск безопасен; runtime-поля PG (`current_task_id`, `cooldown_until`) **не затираются** discovery-sync'ом.

**Тесты:** 11 unit (`test_accounts_sync_unit.py`) + 6 интegration (`test_accounts_sync.py`).

### 1.6. Процесс воркера и Docker (C1, Z4)

#### C1 — `app_balance/queue_worker.py`

| Компонент | Реализация |
|-----------|------------|
| `QueueWorker` | Asyncio main loop: `claim_next` → handler → `complete` / `reschedule_or_fail` |
| `WorkerConfig.from_env()` | `WORKER_ID`, poll interval, lock TTL, filter по `WORKER_TASK_TYPES` |
| Graceful shutdown | SIGTERM/SIGINT → `_stop` Event; **текущая задача дорабатывается** |
| `serve()` | init pool → signal handlers → run → close pool |
| Windows | Fallback `signal.signal()` при отсутствии `add_signal_handler` |
| Release | `finally`: release `task.account_id` если был зарезервирован |

На момент закрытия спринта handler — `_noop_handler` (C1 scope). Инфраструктура release/retry уже на месте для C2.

#### Z4 — Docker Compose

```yaml
queue-worker:
  stop_signal: SIGTERM
  stop_grace_period: 30s
  restart: unless-stopped
  command: ["python", "-m", "app_balance.queue_worker"]
```

- **`queue-worker`** — подключение к prod/staging PG через `.env`
- **`queue-worker-local`** — profile `local`: PG 16 + migrate + worker, полный dev loop без внешних зависимостей

## 2. Текущий спринт S2→S3 (18 июня — ~3 июля) — 5–6 минут

### 2.1. Цель текущей итерации

Две связанные цели:

1. **Закрыть хвост S2** — worker с **полным dispatch** и mock-адаптером.
2. **Пройти S3 (MVP)** — **add-channels через PostgreSQL** от API до реального Telethon на staging.

**Финальный образ через ~2 недели:**

```
POST /add-channels (USE_PG_QUEUE=true)
    → N задач в PG
    → воркер: reserve → check resource → Telethon
    → канал слушается на назначенной сессии
```

### 2.5. Что будет готово по завершении текущего спринта

#### Технически

- Воркер **полностью по ТЗ §5.8**: reserve, resource check, postpone, retry, release

## 5. Заключение — 30 секунд

### Прошлый спринт (S1 + начало S2)

За неделю заложен **production-grade фундамент** PostgreSQL-очереди:

- **~540 строк DDL** с 20+ индексами и 8 monitoring VIEW
- **19 Telethon-op** в seed с реалистичными RPH-лимитами
- **7 Python-модулей** typed async-репозиториев
- **~92 автотеста** (80 PG + 12 SessionClump)
- Docker-воркер с graceful SIGTERM
- Sync аккаунтов discovery → PG
- SessionClump: точечный add/move для воркера

- **~40% инфраструктуры MVP** — готово и протестировано
- **6 критериев §30 ТЗ** — закрыты полностью
- **0% бизнес-сценария** «add-channel через PG» — dispatch и adapter в текущем спринте

### Текущий спринт (S2→S3)

Превращает инфраструктуру в **рабочий продукт**: dispatch по правилам ТЗ + первый production-ready cutover add-channels.

После него можно **включать PG-очередь на staging под флагом** и собирать обратную связь, не дожидаясь продюсеров и полного cleanup legacy (S4–S5).
